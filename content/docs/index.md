Gandalf UI is the internal component library of **Unergy**. It sits on top of [shadcn/ui](https://ui.shadcn.com/) (which itself wraps [Reka UI](https://reka-ui.com/)) and extends it with semantic colors, opinionated variants, and composable primitives designed to be used across the platform.

The goal is not to reinvent a component library from scratch — it is to give every dev a consistent, well-typed, flexible set of building blocks so the platform looks and behaves the same everywhere.

---

## Architecture

The project is organized in clearly separated layers. **Do not mix concerns between layers.**

```
app/components/
  ui/             → Raw shadcn/ui primitives. Do not edit these.
  gandalf/
    base/         → Atomic G-components. Generic, single-responsibility.
    kit/          → Composed components built from base/ primitives.
  blocks/         → App-level compositions. No business logic.
  features/       → Feature-specific UI (knows about domain entities).
```

### `ui/` — Primitives

Generated by shadcn/ui. **Never edit these files.** If you need to change behavior, wrap the component in `gandalf/base/` instead.

### `gandalf/base/` — Atomic Design System

The atomic layer of Gandalf UI. Each component here must:

- Be **generic** — it cannot know about Unergy's business domain (invoices, clients, contracts, etc.)
- Be **single-responsibility** — one concern per component
- Be **composable** — prefer slots over rigid internal layout
- Use **CVA** (`class-variance-authority`) for variants
- Follow the naming convention `G<ComponentName>` (e.g. `GBadge`, `GTabs`)
- Export everything from an `index.ts` in its folder
- **Wrap `ui/` components, not Reka UI directly** — always import from `@/components/ui/<name>`, never from `reka-ui`

### `gandalf/kit/` — Composed Components

Higher-order components composed from `gandalf/base/` primitives. Use this layer for reusable patterns that combine multiple base components into a single, opinionated unit — for example, a checkbox with a card wrapper, or a field with built-in validation display.

A `kit` component should:
- Only import from `gandalf/base/`, never from `ui/` directly
- Still be generic (no business domain knowledge)
- Have a clear, focused API that hides internal composition complexity

### `blocks/` — Composed UI

Compositions of `gandalf/` components that represent a UI pattern used in multiple places across the platform, but without business logic. Example: a search input with a clear button, a page header with breadcrumbs.

A `block` should only be created when the same composition repeats **at least 3 times** across the codebase and a slot-based approach is not ergonomic enough. If a slot solves it, use it — don't create the block.

### `features/` — Feature UI

Components that know about platform-specific data: types like `Factura`, `Cliente`, `Contrato`. These live here and **not** in `gandalf/` or `blocks/`.

## Design Decisions

### Composition over abstraction

Gandalf components expose **slots** rather than locking you into a fixed layout. For example, `GDropdown` exposes `{ open, selectedOption }` via its trigger slot so you can render any element as the trigger:

```vue
<GDropdown v-model="selected" :options="options">
  <template #default="{ open, selectedOption }">
    <GBadge class="cursor-pointer gap-1">
      {{ selectedOption?.label ?? 'Select' }}
      <ChevronDown :class="{ 'rotate-180': open }" />
    </GBadge>
  </template>
</GDropdown>
```

This is always preferred over creating a `BadgeDropdown` component that only handles one specific case.

### Props flow via provide/inject for compound components

When a parent component needs to share state with its children without requiring repetitive prop passing, use Vue's `provide`/`inject`. See `GTabsList` → `GTabsTrigger` as a reference implementation: the `variant` and `align` are provided once on the list and consumed automatically by each trigger.

### `v-model` on all selectable components

Every component with selectable state must support `v-model` via `modelValue` prop + `update:modelValue` emit. Additionally emit a `select` event with the full object for cases where extra metadata is needed.

### Strict typing on the `Option` type

```ts
type Option = {
  label: string
  value: string | number
  disabled?: boolean
  color?: `#${string}`   // hex only, enforced by the type
}
```

## Adding a New Component

Follow these steps every time you add a component to `gandalf/`:

**1. Create the folder**
```
app/components/gandalf/<component-name>/
  G<ComponentName>.vue
  index.ts
```

**2. Wrap the shadcn primitive — don't use Reka UI directly**

`gandalf/` components always import from `@/components/ui/`, never directly from `reka-ui`. The `ui/` layer is the only one that talks to Reka UI. This way, if shadcn changes something internally, only `ui/` needs to be updated.

```vue
<!-- ✓ Correct -->
import { MyComponent } from '@/components/ui/my-component'

<!-- ✗ Wrong -->
import { MyComponent } from 'reka-ui'
```

```vue
<!-- GMyComponent.vue -->
<script setup lang="ts">
import type { MyComponentProps } from 'reka-ui'
import type { HTMLAttributes } from 'vue'
import { reactiveOmit } from '@vueuse/core'
import { MyComponent } from '@/components/ui/my-component'
import { cn } from '@/lib/utils'
import { gandalfMyComponentVariants } from './index'

const props = defineProps<MyComponentProps & {
  class?: HTMLAttributes['class']
  variant?: 'default' | 'outline'
}>()

const delegatedProps = reactiveOmit(props, 'class', 'variant')
</script>

<template>
  <MyComponent
    v-bind="delegatedProps"
    :class="cn(gandalfMyComponentVariants({ variant }), props.class)"
  >
    <slot />
  </MyComponent>
</template>
```

**3. Define variants with CVA in `index.ts`**
```ts
import { cva } from 'class-variance-authority'
import type { VariantProps } from 'class-variance-authority'

export { default as GMyComponent } from './GMyComponent.vue'

export const gandalfMyComponentVariants = cva('base-classes', {
  variants: {
    variant: {
      default: '...',
      outline: '...',
    },
  },
  defaultVariants: { variant: 'default' },
})

export type GandalfMyComponentVariants = VariantProps<typeof gandalfMyComponentVariants>
```

**4. Use `reactiveOmit` for all custom props**

Always omit your custom props (`variant`, `color`, `class`, etc.) before spreading with `v-bind` so they don't leak to the underlying HTML element.

**5. Add a doc page**

Create `content/docs/<component-name>.md` following the same structure as the existing docs. Update `content/docs/index.md` to include the component in the table.

## Conventions

| Rule | Example |
|---|---|
| Component names prefixed with `G` | `GBadge`, `GDropdown`, `GTabs` |
| Atomic components live in `gandalf/base/` | `gandalf/base/badge/GBadge.vue` |
| Composed components live in `gandalf/kit/` | `gandalf/kit/CheckboxCard.vue` |
| Always export from `index.ts` | `export { default as GBadge } from './GBadge.vue'` |
| CVA for all variants | `gandalfBadgeVariants = cva(...)` |
| `reactiveOmit` before `v-bind` | `reactiveOmit(props, 'class', 'variant')` |
| `v-model` on selectable state | `modelValue` + `update:modelValue` |
| Slots over rigid layouts | expose `open`, `selectedOption`, etc. |
| No business logic in `gandalf/` | No `Factura`, `Cliente` types here |
